# UI/UX 设计

> 文档编号：04-UI-UX设计-uiux_design.md | 版本：1.4 | 日期：2026-03-04
> 依赖文档：overview.md（功能列表）, detailed_design.md（API 接口）

## 1. 设计原则

- **用户端**：移动优先（Mobile-first）响应式设计，支持手机/平板/桌面
- **管理端/代理端**：桌面优先，最小适配 1280px 宽度
- **组件库**：Tailwind CSS 4 + shadcn/ui（基于 Radix UI）
- **状态管理**：Zustand（轻量级、immutable-friendly）
- **表单**：React Hook Form + Zod 验证
- **图标**：Lucide React

## 2. 用户端（User Portal）

### 2.1 页面结构与路由

```
/                              → 首页（游戏大厅）
/(auth)/login                  → 登录
/(auth)/register               → 注册
/wallet                        → 钱包概览（余额 + Tab: 最近流水 / 申请记录）
/wallet/deposit                → 充值申请
/wallet/withdraw               → 提现申请
/wallet/transactions           → 完整流水列表
/wallet/bank-cards             → 银行卡管理
/games                         → 游戏列表（同首页，按子平台筛选）
/games/[gameId]/launch         → 游戏启动（转入金额选择）
/orders                        → 订单历史
/orders/[orderId]              → 订单详情
/bet-records                   → 投注记录（来源：厂商 API 同步）
/profile                       → 个人资料（含密码修改入口）
```

## 3. 管理端 + 代理端（Admin/Agent Portal）

### 3.1 页面结构与路由

**管理端**：

```
/(auth)/login                          → 登录
/admin/dashboard                       → 管理员仪表盘

/admin/logs/queries                    → 轮询日志
/admin/logs/actions                    → 操作日志

/admin/reconciliation                  → 对账

```

```
游戏大厅

  → 点击游戏卡片 "进入"
  → 游戏启动页（显示游戏信息 + 金额输入）
  → 输入转入金额（或点击快捷金额）
  → 点击 "进入游戏"
  → 前端自动检测设备类型（PC → ingress=device1, 移动端 → ingress=device2）
  → 检查是否有进行中的游戏
     → 有: Error Toast "已有进行中的游戏，请先退出"
  → Loading 状态（显示 "正在转入资金..."）
  → 成功: 新窗口打开游戏 URL + 页面切换为"游戏进行中"状态 + Toast "已转入 ¥XXX"
  → 失败: Error Toast（"余额不足" / "厂商暂不可用" / "金额超出限额"）
```

### 4.2 充值申请交互

```
钱包页
  → 点击 "充值"
  → 充值申请页
  → 查看收款银行信息
  → 用户线下完成转账
  → 填写金额 + 上传转账凭证
  → 提交
  → Toast "充值申请已提交，等待审核"
  → 状态跟踪（可在钱包页查看申请状态）
  → SignalR 通知: 审核通过 → 余额自动更新 + Toast
  → SignalR 通知: 审核拒绝 → Toast 提示
```

### 4.3 管理员充值审核交互

```
充值审核列表
  → 查看申请详情（金额、凭证图片）
  → 点击 "通过" 或 "拒绝"
  → 确认弹窗（二次确认，防止误操作）
  → 可选填写备注
  → 确认
  → 通过: 自动充值到用户钱包 + 生成流水 + 记录 AdminActionLog
  → 拒绝: 更新申请状态 + 记录 AdminActionLog
```

### 4.4 游戏退出交互

```
方式 1: 用户在平台页面点击 "退出游戏" 按钮
方式 2: 用户直接关闭游戏窗口 → 前端自动检测

检测机制（多层兜底）:
  → 游戏启动后，平台页面显示 "游戏进行中" 状态 + "退出游戏" 按钮
  → Layer 1: setInterval (每 2 秒) 检查 window.closed 属性
  → Layer 2: visibilitychange 事件监听（用户切回平台页面时检查窗口状态）
  → Layer 3: 后端 OrderPollingService 定期检查 Confirmed 超时订单
  → 检测到窗口关闭 → 自动触发退出流程

退出流程:
  → 调用 POST /api/games/exit { orderId }
  → Loading "正在转出资金..."
  → 成功: 余额更新 + Toast "已转回 ¥XXX"
  → 失败: Error Toast + 提示联系客服
  → 兜底: 即使前端未检测到关闭，OrderPollingService 会在后台定期检查 Confirmed 订单
```

## 7. 状态管理

### 7.1 Zustand Stores

```
stores/
├── useAuthStore           # 用户认证状态（user, token, role）
├── useWalletStore         # 钱包状态（balance, frozenBalance, availableBalance, 通过 SignalR 实时更新）
├── useNotificationStore   # 通知状态（未读数、通知列表）
└── useUIStore             # UI 状态（sidebar 折叠、loading 等）
```

### 7.2 SignalR 集成

- 登录后自动建立 SignalR 连接
- 接收到 `BalanceUpdated` → 更新 `useWalletStore.balance`
- 接收到 `OrderStatusChanged` → 触发页面刷新或 Toast
- 接收到 `DepositRequestUpdated` → 充值申请状态变更 Toast（用户端）
- 接收到 `WithdrawRequestUpdated` → 提现申请状态变更 Toast（用户端）
- 接收到 `SystemNotification` → 显示系统通知
- 断线自动重连（指数退避）

## 10. 错误状态处理

所有页面统一的错误状态处理：

| 场景 | 处理方式 |
|------|----------|
| 网络错误 | 全局 ErrorBoundary 捕获，显示 "网络连接失败，请检查网络" + [重试] |
| 401 Unauthorized | 自动跳转登录页，Toast "登录已过期" |
| 403 Forbidden | 显示 "无权限访问" 占位页 |
| 404 Not Found | 显示 "页面不存在" 占位页 |
| 500 Server Error | Toast "服务器异常，请稍后重试" |
| 空数据 | EmptyState 组件，如 "暂无流水记录" / "暂无游戏" |
| Loading | Skeleton 骨架屏（列表/卡片/表格各有对应样式） |
