## 1. 项目概述

### 1.1 背景

游戏聚合平台 — 面向用户提供多种第三方游戏的接入服务。平台作为中间层，管理用户资金（中心钱包），对接第三方游戏厂商 API，处理订单和结算。

### 1.2 MVP 范围

- 接入 **一家** 三方游戏聚合 API（转账钱包模式，聚合 70+ 子平台如 AG/PG/PP/CQ9 等）
- 支持 **三端**：用户端、代理端（单层）、管理端
- **后台人工充值**，无需对接支付通道
- 核心：资金安全、幂等控制、事务完整性、审计可追溯

### 1.3 核心设计决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 游戏对接模式 | 转账钱包（Transfer Wallet） | 资金转入厂商钱包 → 游戏 → 一键回收回平台 |
| 厂商 API 模式 | 聚合 API + 纯轮询（无回调） | 厂商无 Callback，转账结果通过 transferStatus 轮询确认 |
| 玩家账号映射 | VendorPlayerMappings 表 | 内部 GUID → 厂商 5-11 位字母数字 playerId |
| 代理层级 | 单层代理 | 代理直接管理终端用户 |
| 充值方式 | 后台人工充值 | 管理员手动确认 |
| 游戏启动 | 厂商 URL + 新窗口 | 用户在厂商页面中游戏 |
| 转账粒度 | 用户手动指定金额 | 灵活控制风险 |
| 游戏会话 | 单会话限制 | 同一时间只允许一个 Confirmed 游戏订单 |
| 金额限额 | 后台可配置 | 单笔充值 10~50,000、单笔提现 10~50,000、每日累计提现 100,000 |
| 前端框架 | Next.js 15 App Router | 用户端响应式布局 |
| 三端部署 | 两个项目 | 用户端独立；管理端 + 代理端共用 |
| 数据库访问 | Supabase Auth + EF Core | 混合模式 |

## 2. 模块列表

### 2.1 模块总览

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 用户管理 | 注册/登录/角色/个人资料 | P0 |
| 中心钱包 | 余额管理/流水记录/版本号 CAS | P0 |
| 游戏订单 | 转入/转出/状态机/幂等 | P0 |
| 三方 API 对接 | 玩家创建/转入/转出/一键回收/游戏 URL/轮询 | P0 |
| 游戏投注记录 | 同步厂商投注记录（实时+历史），管理端/代理端查看 | P0 |
| 日志审计 | 轮询日志/操作日志/回调日志（预留） | P0 |
| 异常恢复 | Pending 订单恢复/对账 | P1 |
| 告警系统 | 余额不一致/重复结算/签名失败 | P1 |
| SignalR 推送 | 余额变动/提现状态/系统通知 | P1 |
| 代理管理 | 下级用户/报表统计 | P2 |

### 2.2 模块简述

**用户管理** — Supabase Auth 提供注册/登录，平台维护 UserProfile 扩展信息（角色、代理关系、状态等）。角色：User、Agent、Admin。

**中心钱包** — 每用户一个钱包（注册时自动创建），余额使用 decimal，所有变动通过 WalletTransaction 流水。并发控制使用 Version CAS。支持余额冻结（提现申请时预冻结）。余额 = SUM(流水) 必须可校验。充值/提现有金额限额（后台可配置）。

**游戏订单** — 对应一次「转入 → 游戏 → 转出」完整会话。状态机：Pending → Confirmed → Settled / Cancelled / Failed。订单包含 InternalOrderId 和 ExternalOrderId 唯一索引，保证幂等。**单会话限制**：同一用户同一时间只允许一个 Confirmed 状态的订单。

**三方 API 对接** — 抽象 IVendorAdapter 接口，MVP 接入一家聚合 API（70+ 子平台）。支持：玩家创建、余额查询（单平台/一键查询）、转入、转出、一键回收、获取游戏 URL、转账状态查询、游戏列表。**当前厂商无回调机制，全部采用轮询模式**。IVendorAdapter 保留回调接口定义以备未来扩展。需维护玩家 ID 映射（内部 GUID ↔ 厂商 playerId）。

**游戏投注记录** — 后台定期同步厂商投注记录（实时 10 分钟窗口 + 历史按时间段），存储到本地 GameBetRecords 表。管理端查看全局投注记录，代理端查看下级投注记录。

**日志审计** — 所有日志表不可物理删除。VendorCallbackLogs 预留（当前厂商无回调），VendorQueryLogs 记录轮询/转账状态查询结果，AdminActionLogs 记录后台操作。

**异常恢复** — BackgroundService 定期扫描 Pending 状态订单，尝试恢复处理。幂等设计保证重复执行安全。

**告警系统** — 监控余额不一致、重复结算、签名失败等异常事件，记录到 AdminActionLogs 并触发通知。

**SignalR 推送** — 余额变动、提现状态、系统通知推送到前端。禁止前端直接触发资金变动。

**代理管理** — 单层代理查看下级用户列表、余额、订单、统计报表。

## 3. 三端功能概要

### 3.1 用户端（User Portal）

| 功能 | 说明 |
|------|------|
| 注册/登录 | Supabase Auth（邮箱/手机） |
| 密码修改 | 通过 Supabase Auth 修改密码 |
| 个人资料 | 查看/编辑昵称、头像等 |
| 钱包余额 | 查看当前余额、流水历史 |
| 充值申请 | 提交充值申请（线下转账凭证） |
| 提现申请 | 提交提现请求 |
| 银行卡管理 | 添加/删除/设置默认银行卡（最多 5 张） |
| 申请记录 | 查看充值/提现申请状态（钱包页内嵌 Tab） |
| 游戏大厅 | 浏览厂商提供的游戏列表 |
| 进入游戏 | 指定金额转入 → 获取游戏 URL → 新窗口打开（单会话限制） |
| 退出游戏 | 一键回收厂商所有平台余额 → 余额回到平台钱包 |
| 订单历史 | 查看游戏订单记录和状态 |
| 投注记录 | 查看个人游戏投注明细（来源：厂商 API 同步） |
| 实时通知 | SignalR 接收余额变动、系统公告 |

### 3.2 代理端（Agent Portal）

| 功能 | 说明 |
|------|------|
| 登录 | Supabase Auth + Agent 角色 |
| 仪表盘 | 下级用户数、总充值/提现/投注/盈亏概览 |
| 下级用户列表 | 查看直属用户及其状态 |

| 下级投注记录 | 查看下级用户投注明细（按平台筛选） |

### 3.3 管理端（Admin Portal）

| 功能 | 说明 |
|------|------|
| 登录 | Supabase Auth + Admin 角色 |
| 用户管理 | 查看/搜索/禁用/启用用户 |
| 钱包调整 | 人工充值/扣款（生成流水 + 日志） |
| 订单管理 | 查看/搜索订单、人工修改状态（日志）、强制退出用户游戏 |

| 告警查看 | 查看系统告警记录 |

## 4. 模块依赖关系

```
用户管理 ←─── 中心钱包 ←─── 游戏订单 ←─── 三方API对接
   ↑              ↑              ↑              ↑
   └──────────────┴──────────────┴──────── 日志审计
                   ↑              ↑
                   └──────── 异常恢复/对账
                   ↑
                SignalR推送
```

依赖方向：
- **中心钱包** 依赖 用户管理（每用户一个钱包）
- **游戏订单** 依赖 中心钱包（下单需扣款/加款）
- **三方 API 对接** 依赖 游戏订单（更新订单状态）
- **日志审计** 贯穿所有模块
- **异常恢复** 依赖 游戏订单 + 中心钱包
- **SignalR 推送** 依赖 中心钱包（余额变动通知）

8. 充值申请/提现申请流程

### Phase 3：游戏订单 + 三方对接（P0）

12. 玩家创建/转入/一键回收/游戏 URL 流程
13. transferStatus 轮询 + 异常恢复
14. 幂等控制和并发测试
15. GameBetRecord + 投注记录同步服务

### Phase 4：管理后台（P0）

16. 管理端用户管理
17. 钱包手动调整（充值审核/提现审核）
18. 订单管理 + 投注记录查看
19. 日志查看

### Phase 5：告警 + 恢复 + 推送（P1）

20. 异常恢复 BackgroundService

### Phase 6：代理端 + E2E（P2）

## 6. 功能流程概要

### 6.1 游戏订单流程

```
用户选择游戏 → 指定转入金额
  → 检查是否有进行中的游戏（Pending/Confirmed 订单）→ 有则拒绝
  → 检查/创建厂商玩家账号（VendorPlayerMappings）
  → 创建 Pending 订单
  → 钱包扣款（CAS）
  → 调用厂商转入 API → 非明确成功/失败时轮询 transferStatus
  → 获取游戏 URL（含 platType/gameType/ingress/returnUrl）→ 订单 Confirmed
  → 用户在新窗口游戏
  → 用户退出
  → 调用厂商一键回收 API（transferAll）
  → 钱包入账（CAS）
  → 订单 Settled
  → 流水记录 + SignalR 通知
```

### 6.2 人工充值流程

```
用户提交充值申请（金额 + 凭证）
  → 创建 Pending DepositRequest
  → 管理员审核
  → 审批通过：钱包 Credit（CAS） + 流水 + AdminActionLog
  → 审批拒绝：更新申请状态 + AdminActionLog
  → SignalR 通知用户
```

### 6.3 提现申请流程

```
用户提交提现申请（金额 + 银行信息）
  → 验证可用余额（Balance - FrozenBalance）充足
  → 钱包冻结余额 Freeze(amount)（CAS 更新）
  → 创建 Pending WithdrawRequest
  → 管理员审核
  → 审批通过：钱包 ConfirmFrozenDebit(amount)（从冻结余额扣款）+ 流水 + AdminActionLog
  → 审批拒绝：钱包 Unfreeze(amount)（解冻）+ 更新申请状态 + AdminActionLog
  → 管理员线下打款后：标记 Complete
  → SignalR 通知用户
```

### 6.4 钱包手动调整流程

```
管理员选择用户 → 指定操作（充值/扣款/调整）
  → 钱包 Credit/Debit（CAS）
  → 生成 WalletTransaction 流水
  → 记录 AdminActionLog
  → SignalR 通知用户
```

## 7. 风险点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 转入成功但转出失败 | 资金滞留厂商 | 异常恢复任务自动重试转出 |
| 厂商无回调机制 | 状态更新延迟 | 纯轮询模式 + transferStatus 状态查询 + 异常恢复任务 |
| 厂商 API 限流 | 请求被拒绝(10009) | 客户端限流控制 + 重试退避策略 |
| 玩家 ID 映射失败 | 无法调用厂商 API | VendorPlayerMappings 映射表 + 首次游戏时自动创建 |
| 转账非明确结果 | 资金状态不确定 | transferStatus 轮询确认 + 超时后人工处理 |
| CAS 版本冲突高频 | 扣款失败率高 | 重试机制 + 告警监控 |
| 厂商 API 不稳定 | 订单卡在 Pending | 轮询 + 超时 + 人工干预 |
| 对账不一致 | 余额错误 | 告警 + 冻结 + 人工处理 |
| 提现审批前余额不足 | 用户余额已被消费 | 申请时冻结余额（Freeze），审批时从冻结扣款 |
| 用户同时开多个游戏 | 资金分散、恢复复杂 | 单会话限制（LaunchGame 时检查是否有 Confirmed 订单） |
| 单人开发风险 | 关键代码无 Review | AI 辅助 Review + 高测试覆盖 |

## 8. Definition of Done（DoD）

每个功能模块完成标准：

- [ ] Domain 层单元测试存在且通过（覆盖率 >= 90%）
- [ ] Backend 总体测试覆盖率 >= 85%
- [ ] 关键业务流程有集成测试
- [ ] API 端点有集成测试
- [ ] 前端关键流程有 E2E 测试
- [ ] 代码 lint/格式化通过
- [ ] 无硬编码密钥
- [ ] 资金相关变更有幂等校验
- [ ] 状态变更有审计日志
- [ ] CI 全部通过
